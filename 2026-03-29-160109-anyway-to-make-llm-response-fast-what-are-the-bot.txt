  LLM Response Speed ‚ÄĒ Bottlenecks & Fixes

  After reading through brain.ts, here are the bottlenecks ranked by impact:

  1. No Streaming (biggest win)

  brain.ts:554 ‚ÄĒ You're using this.chat.sendMessage() which waits for the entire
   response before returning anything to the user. The Gemini SDK supports
  sendMessageStream() which sends tokens as they arrive.

  Impact: User sees nothing until the full response + all tool calls complete.
  On complex responses this can be 5-15s of blank screen.

  2. System Prompt is Massive (~3K+ tokens rebuilt every call)

  brain.ts:150-295 ‚ÄĒ buildSystemPrompt() embeds the full skill table, MSP
  expertise block, safety rules, communication guide, mobile app docs, AND the
  learner context block + knowledge file on every single turn. This all goes to
  the model as input tokens on every request.

  Fixes:
  - Cache the system prompt and only rebuild when skills/knowledge actually
  change
  - Move static content (MSP expertise, mobile app docs, safety rules) into
  Gemini's systemInstruction config instead of injecting as a fake user message
  - The skill table is ~20 detailed entries ‚ÄĒ consider a shorter version

  3. History Saved as Fake User/Model Messages (wasted tokens)

  brain.ts:626-629 ‚ÄĒ System prompt is injected as a user message with a fake
  model reply. This doubles the token cost vs using Gemini's native
  systemInstruction field. Every turn, the model re-reads the entire system
  prompt as conversation history.

  Fix: Use systemInstruction in the model config:
  return genAI.getGenerativeModel({
    model: modelId,
    systemInstruction: buildSystemPrompt(),
    tools: tools,
  });
  This removes ~3K tokens from the conversation history.

  4. Context Compaction Threshold Too High (800K tokens)

  brain.ts:692 ‚ÄĒ Compaction only triggers at 800K tokens, and the check only
  runs every 20 turns (brain.ts:1358). Long conversations accumulate massive
  context that slows every request linearly.

  Fix: Lower to ~200K-300K. Also check more frequently (every 10 turns).

  5. getHistory() + countTokens() on /status and /export

  These slash commands hit the Gemini API just to count tokens, adding
  unnecessary latency. Cache the token count after each turn instead.

  6. Learner Queued on Every Message

  brain.ts:1521 ‚ÄĒ this.learner.queueAnalysis(this.history) sends the full
  conversation to a separate Gemini model after every turn. While async, it
  competes for API quota and can trigger 429s that affect the main model.

  7. saveHistory() is Synchronous FS Write

  brain.ts:634-647 ‚ÄĒ fs.writeFileSync blocks the event loop on every turn. As
  history grows, this serialization gets slower.

  Fix: fs.promises.writeFile (async).

  8. Tool Definitions Rebuilt on Every Model Creation

  brain.ts:411 ‚ÄĒ getToolDefinitions() is called every time createModel() runs
  (on failover, model switch, refresh). If this function does any I/O or heavy
  computation, it adds up.
